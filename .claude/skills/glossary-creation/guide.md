# 用語集作成ガイドライン

## 用語集の目的

用語集は、プロジェクト内で使用される専門用語を統一し、チーム全体での認識齟齬を防ぐための重要なドキュメントです。新メンバーのオンボーディングにも役立ちます。

## 作成時の原則

### 1. 明確で簡潔な定義

専門家でない人でも理解できる定義を心がけます。

**良い例:**
```
### JWT (JSON Web Token)
**定義:** ユーザーの認証情報を安全に送信するための、
JSON形式でエンコードされたトークン規格
```

**悪い例:**
```
### JWT
JWTはトークンです。
```

### 2. 具体的な使用例を含める

実際のプロジェクトでどう使われるかを示します。

**良い例:**
```
**使用例:** 「ユーザーがログインすると、サーバーはJWTを発行し、
以降のAPIリクエストではこのトークンを使って認証を行う」
```

### 3. 関連用語へのリンク

用語間の関係性を示します。

**良い例:**
```
**関連用語:** [認証](#認証-authentication), [認可](#認可-authorization)
```

## カテゴリ分類

### 推奨カテゴリ

1. **ビジネス用語**: プロダクト固有のドメイン用語
2. **技術用語**: 一般的な技術用語（API, REST等）
3. **ドメイン用語**: ドメインモデルに関する用語
4. **アーキテクチャ用語**: アーキテクチャパターンに関する用語
5. **データベース用語**: データベース関連の用語
6. **セキュリティ用語**: セキュリティに関する用語
7. **開発プロセス用語**: CI/CD等のプロセス用語
8. **テスト用語**: テストに関する用語

### カテゴリ分類のポイント

**適切な分類:**
- 用語は最も関連の深いカテゴリに配置
- 複数のカテゴリに該当する場合は、主要な側面で判断
- 分類に迷ったら、チームで議論して決定

## 用語抽出の方法

### 1. ドキュメントからの抽出

各ドキュメントを読み、専門用語をリストアップ:

**PRDから:**
- ビジネス用語
- ユーザーストーリーに登場する用語
- KPI関連の用語

**機能設計書から:**
- API関連の用語
- データフロー関連の用語
- バリデーション用語

**アーキテクチャ設計書から:**
- アーキテクチャパターン名
- レイヤー名
- デザインパターン名

### 2. 重複の排除

同じ意味で異なる用語が使われている場合は統一:

**例:**
- 「ユーザー」と「利用者」 → 「ユーザー」に統一
- 「ログイン」と「サインイン」 → プロジェクトの方針に従って統一

### 3. 略語の展開

すべての略語は正式名称と一緒に記載:

**例:**
```
### API (Application Programming Interface)
**定義:** ...

## 略語一覧
| API | Application Programming Interface | ... |
```

## 用語定義のフォーマット

### 基本フォーマット

```markdown
### [用語名]
**定義:** 簡潔で明確な定義（1-2文）

**使用例:** 「実際のプロジェクトでの使用例」

**関連用語:** [関連用語1](#リンク), [関連用語2](#リンク)

**英語表記:** English Term（日本語用語の場合）

---
```

### 複雑な用語の場合

追加情報を含める:

```markdown
### エンティティ
**定義:** 一意の識別子を持ち、ライフサイクル全体を通じて
追跡されるオブジェクト

**特徴:**
- 一意の識別子（ID）を持つ
- ライフサイクルを通じて同一性が保たれる
- 可変（状態が変化しうる）

**使用例:** 「Userエンティティは、ユーザーの名前や
メールアドレス等の属性を持ち、IDで識別される」

**対比:**
- [値オブジェクト](#値オブジェクト): IDを持たず、値で識別される

**関連用語:** [DDD](#ddd-domain-driven-design),
[値オブジェクト](#値オブジェクト)

---
```

## プロジェクト固有の用語

### ビジネスドメインの用語

プロジェクト固有の用語は特に丁寧に説明:

**例（教育システムの場合）:**
```markdown
### 講師
**定義:** プラットフォーム上でコースを作成し、
生徒に授業を提供するユーザー

**権限:**
- 自分のコース内容の編集
- 生徒の進捗確認
- 課題の採点

**使用例:** 「講師は新しいコースを作成し、
そのコース内で複数のレッスンを公開できる」

**関連用語:** [生徒](#生徒), [コース](#コース), [レッスン](#レッスン)

---

### 生徒
**定義:** プラットフォーム上でコースを受講するユーザー

**権限:**
- コースの閲覧・受講
- 課題の提出
- 講師への質問

**使用例:** 「生徒はコースに登録することで、
レッスン動画を視聴し、課題を提出できる」

**関連用語:** [講師](#講師), [コース](#コース)

---
```

## 略語一覧の作成

### テーブル形式

```markdown
## 略語一覧

| 略語 | 正式名称 | 日本語 |
|------|----------|--------|
| API | Application Programming Interface | アプリケーションプログラミングインターフェース |
| REST | Representational State Transfer | レプレゼンテーショナルステートトランスファー |
| CRUD | Create, Read, Update, Delete | 作成、読取、更新、削除 |
```

### ソート順

- アルファベット順にソート
- 日本語の用語は五十音順にソート

## 用語集のメンテナンス

### 更新タイミング

- 新しい用語がドキュメントに追加された時
- 既存の用語の定義が変更された時
- チームで用語の使い方に関する議論があった時

### 更新履歴

変更を追跡:

## リンクの設定

### アンカーリンク

マークダウンのアンカーリンクを活用:

```markdown
### API (Application Programming Interface)
**関連用語:** [REST](#rest-representational-state-transfer)

---

### REST (Representational State Transfer)
**関連用語:** [API](#api-application-programming-interface)
```

### リンクの命名規則

- 英語用語: `#term-full-name`（例: `#api-application-programming-interface`）
- 日本語用語: `#日本語そのまま`（例: `#エンティティ`）
- スペースはハイフンに置換

## チェックリスト

用語集作成完了前に以下を確認:

- [ ] すべての専門用語が含まれている
- [ ] 各用語に明確な定義がある
- [ ] 使用例が記載されている
- [ ] 関連用語へのリンクが設定されている
- [ ] カテゴリ分類が適切である
- [ ] 略語一覧が含まれている
- [ ] プロジェクト固有の用語が詳しく説明されている
- [ ] 同じ意味の用語が統一されている
- [ ] リンクが正しく機能する
- [ ] 更新履歴が記載されている

## よくある間違い

### 1. 定義が曖昧

**悪い例:**
```
### API
外部とのやり取りをするもの
```

**良い例:**
```
### API (Application Programming Interface)
**定義:** アプリケーション間でデータや機能を
やり取りするためのインターフェース仕様
```

### 2. 使用例がない

定義だけでなく、必ず使用例を含める。

### 3. 関連用語のリンク切れ

リンク先の用語が存在することを確認。

### 4. カテゴリの重複

同じ用語が複数のカテゴリに登場しないようにする。

### 5. 専門用語で専門用語を説明

**悪い例:**
```
### ORM
O/Rマッパーのこと
```

**良い例:**
```
### ORM (Object-Relational Mapping)
**定義:** オブジェクト指向プログラミング言語の
オブジェクトとリレーショナルデータベースのテーブルを
自動的に変換する技術

**使用例:** 「ORMを使うことで、SQLを直接書かずに
データベース操作をオブジェクト操作として記述できる」
```

## 用語の優先順位

すべての用語を網羅するのは困難なため、優先順位をつける:

### 高優先度
1. プロジェクト固有のビジネス用語
2. 頻繁に使用される技術用語
3. 混乱しやすい用語（類似の用語が複数ある場合）

### 中優先度
1. アーキテクチャパターン用語
2. データベース用語
3. セキュリティ用語

### 低優先度
1. 一般的なプログラミング用語（変数、関数等）
2. 広く知られている技術用語

## 用語集の活用方法

### 新メンバーのオンボーディング

- 用語集を最初に読んでもらう
- 不明な用語があれば追加してもらう

### ドキュメント作成時

- 新しい用語を使う場合は用語集に追加
- 既存の用語は用語集の定義に従う

### レビュー時

- 未定義の専門用語が使われていないか確認
- 用語の使い方が定義と一致しているか確認
